—— THE PROBLEM

Raw YAML Doesn't Scale

Every Kubernetes deployment starts with YAML files — Deployments, Services, Routes. Copy them, tweak them, apply them. It works until it doesn't.
Copy-paste drift — each environment gets its own copy, and they diverge silently
No versioning — which YAML is running in production? When did it change?
No rollback story — oc apply is a one-way operation without manual snapshots
—— THE SOLUTION

Helm Is a Package Manager for Kubernetes

A Helm chart bundles all the Kubernetes resources an application needs into a single, versioned, configurable package — with built-in install, upgrade, and rollback semantics.

Without Helm

Manually manage each YAML file. Copy-paste values. Run oc apply -f on each file individually. Hope you got the labels right.

With Helm

One command: helm install. All resources templated, versioned, and deployed as a single atomic release. Rollback with helm rollback.

—— VERSION CONTROL

Charts Are Versioned Artifacts

Every Helm chart carries its own version in Chart.yaml. This makes your deployment configuration a first-class, trackable artifact — just like your application source code.
# Chart.yaml — the chart's identity apiVersion: v2 name: httpd-demo version: 0.1.0 # ← Chart version (SemVer) appVersion: "1.0.0" # ← App version (the image) description: A Helm chart for Apache HTTPD on OpenShift
Store charts in Git — review changes in PRs, tag releases, audit who changed what and when
SemVer built in — major/minor/patch versioning so consumers know when breaking changes land
Chart vs. App version — chart config and application image version evolve independently
—— TEMPLATING

Go Templates Make Charts Dynamic

Helm templates aren't static YAML — they use Go's template engine to inject values, add conditionals, and loop over data at render time.

Value Injection

# deployment.yaml replicas: {{ .Values.replicaCount }} image: "{{ .Values.image.repository }} :{{ .Values.image.tag }}"

Conditionals

# route.yaml {{- if .Values.route.enabled }} apiVersion: route.openshift.io/v1 kind: Route ... {{- end }}
Built-in objects like .Release.Name, .Chart.Name, and .Chart.Version are always available — no extra config needed.
—— VALUES

Values Cascade from Defaults to Overrides

Helm resolves values in a specific priority order. The same chart serves every environment — only the values change.
values.yaml (defaults)
↓ overridden by
-f values-prod.yaml
↓ overridden by
--set key=value
# Use a per-environment values file + CLI overrides helm upgrade my-httpd ./charts/httpd-demo \ -f values-prod.yaml \ --set image.tag=2.4-ubi9
—— REPOSITORIES

Share Charts via Repositories

Charts aren't just local folders — they can be packaged and published to repositories for team-wide or organization-wide reuse.
# Add a public chart repository helm repo add bitnami https://charts.bitnami.com/bitnami helm repo update # Search for available charts helm search repo nginx # Pull a chart from an OCI registry helm pull oci://registry.example.com/charts/my-app --version 1.2.0
Artifact Hub — discover thousands of community charts at artifacthub.io
OCI registries — store charts alongside images in Quay, Nexus, or Harbor
—— DRY RUN

Preview Before You Deploy

Helm lets you render and validate charts locally — no cluster required. Catch mistakes before they hit production.
# Render templates locally — see exactly what Kubernetes will receive helm template my-httpd ./charts/httpd-demo # Render with overrides to test specific configurations helm template my-httpd ./charts/httpd-demo --set replicaCount=5 # Lint the chart for structural issues helm lint ./charts/httpd-demo # Dry-run against the cluster (validates with API server) helm install my-httpd ./charts/httpd-demo --dry-run
—— DEMO · STEP 1

Log In and Own Your Namespace

Each participant works in an isolated OpenShift project. Log in with your own credentials, then create your personal namespace.
# Log into the cluster oc login <cluster-api-url> -u <your-username> # Capture your username for use throughout the demo export OC_USER=$(oc whoami) # Create your own project oc new-project ${OC_USER}-helm-demo # Confirm you're in the right place oc project
—— DEMO · STEP 2

One Command to Deploy Everything

# Install the chart — creates Deployment, Service, and Route helm install my-httpd ./charts/httpd-demo # Watch pods come up oc get pods -w # Grab the route URL and open in browser oc get route my-httpd-httpd -o jsonpath='{.spec.host}'
Open the route URL in your browser — you should see the Apache HTTP Server Test Page, proof that your application is live on OpenShift.
—— DEMO · STEP 3

Override Values Without Editing Files

Helm's --set flag lets you override any value from values.yaml at install or upgrade time — no file edits required.
# Scale to 3 replicas helm upgrade my-httpd ./charts/httpd-demo --set replicaCount=3 # Watch them appear oc get pods -w # Chain multiple overrides helm upgrade my-httpd ./charts/httpd-demo \ --set replicaCount=2 \ --set resources.limits.memory=512Mi
—— DEMO · STEP 4

Every Change Is Tracked and Reversible

Helm tracks every upgrade as a numbered revision. Roll back to any previous state with one command.
# See every revision helm history my-httpd # Roll back to revision 1 (the original install) helm rollback my-httpd 1 # Verify oc get pods helm status my-httpd

—— CLEANUP

Clean Up in One Command

Helm knows every resource it created. Uninstall removes them all atomically — no orphaned Services or stale Routes.
# Remove the entire release helm uninstall my-httpd # Verify nothing remains oc get all # Delete your project oc delete project ${OC_USER}-helm-demo
